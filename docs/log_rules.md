# ログ設計ルール

## 目的

本ドキュメントは、次の 3 点を主目的として、アプリケーションログの設計・実装・運用ルールを定義する。

- 障害対応: 問題発生時に原因調査と影響範囲の特定をしやすくする
- 性能分析: 遅延やタイムアウトの発生箇所を把握しやすくする
- 監査補助: 重要な処理の実行有無を追跡できるようにする

## 対象範囲

- HTTP API のリクエスト処理
- バッチ / 非同期ジョブ
- 外部 API 呼び出しと失敗
- DB アクセスの失敗や遅延
- 認証 / 認可に関わる重要イベント

## 基本方針

- ログは構造化ログで出力する
- ログは人間が読む用途と、機械が集計・検索する用途の両方を考慮する
- 1 リクエスト / 1 ジョブにつき、少なくとも 1 つのサマリログを残す
- 原則としてアプリケーションは `stdout` に出力し、収集・転送・保存は実行基盤側で行う
- 後から検索・集計しやすいように、フィールド名は固定化する

## 出力粒度

最低限、次の粒度で出力する。

### HTTP リクエスト

- `http_request_succeeded`
  - リクエストが正常終了したとき
- `http_request_rejected`
  - バリデーションエラー、認証エラー、認可エラーなどで処理を拒否したとき
- `http_request_failed`
  - 未処理例外やタイムアウトやぬるぽなどで 5xx にするとき

### 外部依存

- `external_api_succeeded`
  - 外部 API 呼び出しが成功したとき
- `external_api_failed`
  - 外部 API 呼び出しが失敗したとき
- `db_query_failed`
  - DB エラーが発生したとき

### バッチ / 非同期処理

- `job_started`
  - ジョブ開始時
- `job_completed`
  - ジョブ正常終了時
- `job_failed`
  - ジョブ異常終了時

### セキュリティ / 監査補助

- `login_succeeded`
  - ログイン成功時
- `login_failed`
  - ログイン失敗時
- `permission_denied`
  - 権限不足で操作が拒否されたとき

`debug` レベルの一時的な詳細ログは、恒久的なログ種別としては扱わない。

## 共通スキーマ

全ログで次の共通フィールドを持つことを基本とする。

| フィールド | 必須 | 説明 |
| --- | --- | --- |
| `level` | 必須 | `debug` / `info` / `warn` / `error` |
| `component` | 必須 | API, batch, worker などのコンポーネント名 |
| `job_id` | 必須/バッチ以外はnull可 | HTTP リクエスト単位の相関 ID |
| `request_id` | 必須/HTTP以外はnull可 | HTTP リクエスト単位の相関 ID |
| `user_id` | 必須/未ログイン時はnull可 | 利用者を識別できる内部 ID |
| `http_status_code` | 任意 | HTTP ステータスコード |
| `message` | 必須 | 人間向けの短い説明 |
| `stack_trace` | 任意 | スタックトレース |

スタックトレースの扱い:

- 未処理例外では出力を検討する
- 想定内エラーでは冗長なスタックトレースを避ける
- スタックトレース内に秘匿情報が含まれないように注意する

## ログレベル方針

- `debug`
  - ローカル開発や一時調査用
  - 本番では常時有効にしない
- `info`
  - 正常系の主要イベント
  - リクエスト完了、ジョブ開始 / 完了、重要操作の成功
- `warn`
  - 即時障害ではないが、異常の兆候として追跡したいもの
  - リトライ可能な外部 API 失敗、想定内のフォールバック、遅い処理
- `error`
  - ユーザー影響がある失敗、または運用対応が必要なもの
  - 未処理例外、タイムアウト、ジョブ失敗、外部依存の継続的失敗

4xx 系の扱いは次を基本とする。

- 想定内の入力エラーや認証失敗は `info`
- 実装不備や調査対象になる異常な 4xx は `warn`

## 相関ルール

- HTTP リクエストでは `request_id` を必須とする
- バッチ / 非同期処理では `job_id` を必須とする

## 秘匿情報 / PII ルール

次の情報はログ保存を禁止する。

- メールアドレス
- 電話番号
- 住所
- パスワード
- トークン
- API キー
- Cookie
- Authorization ヘッダ
- セッション ID
- 個人情報を含むリクエストボディ全文

必要な場合は次のいずれかで対応する。

- マスクする
- 件数や有無など、識別できない形に要約する

## 出力先 / 保持 / 閲覧

- アプリケーションは JSON 形式で `stdout` に出力する
- ログの保管・検索は AWS CloudWatch に集約する
- 保持期間は運用要件とコストを踏まえてインフラ側のリポジトリで別途定義する。(個人事業主のため所得税法に基づき現時点では7年を想定)
- 少なくとも AI API を呼び出したログは監査用途で保持する
- 本番ログの閲覧権限は必要最小限に限定する

## 出力量の制御

- 正常系はサマリログを中心に出力する
- ループ内や高頻度処理の詳細ログは原則 `debug` に限定する
- 同一内容の大量出力は避ける
- 大きなペイロードはそのまま出さず、サイズや件数に要約する

## 運用ルール

- 新しいログ項目を追加する際は、利用目的を明確にする
- フィールド名は共通スキーマで統一する。
- 個人情報や秘匿情報が含まれないこと
- 一時調査用の詳細ログは、調査後に削除または `debug` に戻す
- 互換性を壊すフィールド変更は影響範囲を確認してから行う
- 必須フィールドが欠けていないこと
- 異常系で原因調査に必要な情報が不足していないこと
- ログ量が過剰になっていないこと
